## NEW findings (code added / changed 2026-07-13 → 2026-07-14)

### NEW-1 — metric-catalog `applicable` flag marks a placeholder float `0.0` as a real value (Medium)
- **Where:** `engine/metric_catalog.py:192-200` (the `_RIBBON_ENTRIES` loop constructs `CatalogRow`
  without passing `applicable`, so it defaults `True`); source values from `engine/metrics/ribbon.py:216-217`
  (`avg_float = round(...) if floats else 0.0`, `max_float = ... if floats else 0.0`).
- **What:** ADR-0219 (audit L1) added `applicable` so an unmeasurable Workbench cell renders "—" not a
  fabricated `0`. The DCMA half is correct (`applicable = status is not NOT_APPLICABLE`), but the two
  **float** ribbon extras (`avg_float_days`, `max_float_days`) degrade to `0.0` when the incomplete-
  activity float population is empty, yet are emitted `applicable=True`.
- **Failure scenario:** load a schedule at a closeout/final data date where every non-summary activity
  is 100% complete → `floats` is empty → `avg_float_days = max_float_days = 0.0` → the Metric Workbench
  (grid + Excel) shows "Avg Float 0.0 days / Max Float 0.0 days" as a **real mean/max** when the honest
  reading is N/A. This is exactly the "placeholder 0 presented as real" class the flag was introduced to
  prevent — a fidelity regression in a just-shipped fix. (The other extras are fine: `insufficient_detail`
  / `merge_hotspot` are genuine counts where 0 is real; `logic_density` only zeroes on an empty network.)
- **Fix direction (future PR):** for the two float extras, set `applicable` from whether the incomplete-
  float population is non-empty (e.g. carry an "N incomplete" flag off `compute_ribbon`, or mark
  `avg/max_float_days` NA when the population is 0). Pin with an all-complete fixture.

### NEW-2 — margin-dashboard erosion fit can mix measurement bases across versions (Low)
- **Where:** `engine/margin_dashboard.py:55-62` (`_finish_offset` falls back to `cpm.project_finish`
  when `target_uid` is absent from a version) + `:_erosion` (fits `effective_margin_wd` vs status date
  across all versions regardless of basis).
- **What:** margin is measured to the target milestone when present, else the project finish (the
  documented operator choice). But the erosion least-squares fit treats all versions' `effective_margin_wd`
  as one series. If the target exists in some versions and is absent (deleted/renamed) in others, the
  series mixes margin-to-target with margin-to-project-finish.
- **Failure scenario:** target M in v1/v2 (margin-to-M ≈ 10 wd), M deleted in v3 (project-finish margin
  ≈ 60 wd) → the fit sees margin *rising* → a bogus near-zero/positive erosion rate and a wrong (or no)
  zero-margin date, even though margin-to-M is actually eroding. Per-version burn-down figures are still
  each correct; only the *trend* is polluted.
- **Fix direction (future PR):** fit the erosion only over versions that share one basis (e.g. only those
  where the target resolved), and disclose when versions were excluded; or surface a "mixed basis"
  warning. Low because it needs a target present-then-absent across the loaded set.

### NEW-3 — air-gap static-scan test omits the new pages + JS (Low; extends prior L5)
- **Where:** `tests/web/test_airgap.py:49-85` (hard-coded page + static-asset lists).
- **What:** the scan's page list stops before `/scorecards` (`app.py:3903`) and `/margin` (`:2574`),
  and its static-asset list runs `app.js…sra.js`, omitting the 3 new files `drilldown.js`,
  `margin_dashboard.js`, `scorecards.js` (and the pre-existing omissions /evm /resources /cei /trend
  /volatility /path /workbench /gantt /mission /compare /brief /risks).
- **Why it is Low, not live:** the runtime CSP (`default-src 'self'; connect-src 'self'; img-src 'self'
  data:`) is added to **every** response by the global middleware, so a browser blocks any remote asset
  on these pages regardless; and the 3 new JS files are clean today (no external reference). This is a
  test-coverage gap (a future CDN `<script>` on these pages would go uncaught), not an active hole.
- **Fix direction (future PR):** enumerate `app.routes` instead of a hard-coded list (the prior L5 fix),
  which subsumes this.
