## Findings in detail

### C1 — [CRITICAL] Save→reopen (.json) silently drops 11+ forensic fields — VERIFIED
`importers/json_schedule.py:180` (`to_json_text`), `:95` (`_task`), `:75` (`_calendar`).

`to_json_text` is the **Save .json** path (`web/app.py:1156`). Round-tripping a maximal `Schedule` through
`parse_json_text(to_json_text(sch))` and diffing every field shows these silently lost:

```
Task:     calendar_uid 42→None, outline_level 3→0, is_estimated_duration True→False,
          is_level_of_effort True→False, is_active False→True,
          physical_percent_complete 33.0→None, stored_total_float_minutes -240→None,
          stored_is_critical True→None, custom_fields (...)→()
Schedule: custom_field_labels ('CA-WBS','Risk')→()
Calendar: working_days (2025-01-18,)→(), day_segments ((480,720),(780,1020))→()
```

Why it is **Critical**: `stored_total_float_minutes` / `stored_is_critical` are the Acumen-parity stored
values that `_common.effective_total_float` / `is_effective_critical` **prefer** over recomputed CPM float.
Dropping them silently degrades DCMA float metrics on a re-opened progressed file — a **Law-2 fidelity break,
invisible to the user**. `is_active` even flips default (False→True), silently re-activating an inactive
task; `custom_fields`/`custom_field_labels` empty the group/filter population; `calendar.working_days`/
`day_segments` are the SSI driving-slack parity inputs. The existing round-trip tests assert only a subset,
so the loss is unguarded; the `except ImporterError: raise` ordering means a valid-but-lossy friendly doc
never reaches the strict fallback, so it is silent, not an error.

### C2 — [CRITICAL] Default interpretive Ask-the-AI passes model-invented numbers to the operator — VERIFIED
`ai/qa.py:393` (gate runs only `if mode != "interpretive"`), default `qa_mode="interpretive"` (`ai/backend.py:53`).

The charter/docstrings claim "every AI-emitted number is re-verified against engine citations so the model
can never introduce an unsourced number." That is **false in the default mode**: in interpretive mode the
figure-subset gate is skipped and the raw model text is returned. The repo's own test pins it:

```python
# tests/web/test_ask_everywhere.py:161
assert body["answer"] == "That derives to 31415 days." and body["mode"] == "interpretive"
```

`31415` is a number the engine never computed; it reaches the client `answer` field. The only mitigation is
a UI disclaimer plus the cited facts shown alongside. In a testimony context an analyst can copy a fabricated
figure believing the "no unsourced number" guarantee holds. This is an intentional M18 design choice, but it
**contradicts a stated invariant** — the most consequential gap relative to the charter's framing.
Resolution requires an **operator decision** (default to strict / annotate AI-derived figures inline /
explicitly scope the guarantee to narrative+briefing+strict-Q&A).

### H1 — [HIGH] `_apply_ssi_setup` 500s and half-mutates the session on a non-list `affected` — VERIFIED
`web/app.py:6126`. Route `POST /sra/ssi/load`.

```python
affected = tuple(u for u in item.get("affected", []) if _ok(u))   # no isinstance guard
```

The sibling fields (`overrides_minutes`, `factors`, `bcwc_minutes`) all `isinstance(..., dict)`-guard before
iterating; `affected` does not. `{"affected": 5}` or `{"affected": null}` reaches `for u in <value>` and
raises `TypeError` → **500** (the route's docstring promises a hand-edited file validates cleanly). Worse:
the crash fires **after** `sra_factor_rows`, `sra_focus_uid`, `sra_low/ml/high`, `sra_overrides`,
`sra_factors`, `sra_bcwc` have already been assigned, so a failed load **leaves the session partially
mutated** — every model input replaced except the risk register. (`"affected":"35"` iterates characters →
both `_ok()` false → the risk is silently dropped, a softer variant of the same gap.)

### H2 — [HIGH] The reattach figure-gate guards digits but not prose — VERIFIED
`ai/citations.py:71-99` (`preserves_figures` compares only the `\d+(?:\.\d+)?` multiset).

A rephrase that keeps every number but injects an unsourced, defamatory claim passes the gate:

```
source : "[HIGH/logic] 12 of 126 activities have hard constraints. Review and justify each."
tamper : "The contractor DELIBERATELY CONCEALED schedule fraud: 12 of 126 activities have hard
          constraints, proving intentional manipulation. Review..."   → KEPT (digits 12/126 match)
```

The narrative is described as forensic evidence used in testimony. A local model can attribute intent/fraud,
invert a finding's meaning, or add conclusions the engine never made, and the gate keeps it as long as the
numbers line up. The end-to-end wiring test only checks a fabricated *number* is stripped, never prose.

### H3 / H4 — [HIGH] XER: one malformed id sinks the whole file, even for droppable rows — VERIFIED
`importers/xer.py:168` (`all_task_ids = {_req_int(t,"task_id") for t in all_tasks}`) and `:415-416`
(`_req_int` on `TASKPRED` endpoints *before* the in-scope/dangling/duplicate filter).

The importer's documented posture is that cross-project / out-of-scope / dangling P6 rows are "valid
real-world states… dropped with a count logged." But `all_task_ids` is built with strict `_req_int` over
**every** task row across **all** projects, and `TASKPRED` endpoints are strict-parsed before the drop
logic. Verified: a two-project XER where the *non-selected* project carries `task_id=GARBAGE`, or a single
`TASKPRED` row with `task_id=BADSUCC`, raises `ImporterError` and refuses an otherwise-valid file — a Law-2
tolerance violation on a plausible multi-project export.

### M1 — [MEDIUM] Inactive tasks are never excluded from the CPM or any metric — VERIFIED
`model/task.py:74` (field set by both importers) with **no consumer** anywhere in `engine/`
(`grep -rln is_active src/schedule_forensics/engine/` → nothing). MS Project and Acumen Fuse exclude
inactive tasks from scheduling, rollups, and metric populations. On a real schedule with inactive tasks,
they are counted in DCMA denominators (01/05/06/07/08/10/11, Schedule-Quality, EVM, float bands) and pulled
into the CPM network — diverging from the reference tools. The goldens carry `<IsInactive>0</IsInactive>`
throughout, so the **parity gate cannot catch this**. Resolution needs a behavior decision (exclude vs.
document) plus a new golden containing an inactive task to pin it.

### M2 — [MEDIUM] Pre-commit CUI guard + `.gitignore` omit `.aft` and `.docx` — VERIFIED
`.githooks/pre-commit:13` blocklist: `\.(mpp|mpt|mpx|xer|xml|pmxml|csv|xls|xlsx|pbix|mspdi|pkl|pickle)$`.
CLAUDE.md Law 1 states the guard "blocks `.mpp`/`.xlsx`/`.aft`/`.xer`/`.docx`." Neither the hook regex nor
`.gitignore` actually blocks `.aft` (the NASA "Bible") or `.docx` (a named CUI export). A stray `.aft`/`.docx`
dropped at the repo root for Claude to read is silently committable (even with `git add -f`). Mitigated by
`00_REFERENCE_INTAKE/*` being git-ignored and the operator-confirmed CUI boundary, but the guard does not
deliver what the spec claims and **no test asserts the blocklist contents**, so it drifted undetected.

### M3 — [MEDIUM] Friendly JSON fabricates a `project_start` of 2025-01-06 — VERIFIED
`importers/json_schedule.py:168`: `_dt(data.get("project_start")) or dt.datetime(2025, 1, 6, 8, 0)`.
Missing key / explicit null / empty string all yield a hard-coded `2025-01-06 08:00`. Everywhere else the
codebase treats a missing source date as `None`; MSPDI and XER both **raise** when project start is missing.
The friendly importer instead invents a CPM-anchor date — a fabricated forensic input that also masks
truncated/hand-edited files.

### M4 — [MEDIUM] MSPDI baseline-finish summary filter uses the raw tag, not effective `is_summary` — VERIFIED
`importers/mspdi.py:531` force-summarizes UID-0 (`is_summary=... or uid == 0`), but `_project_baseline_finish`
(`:602`) skips on the raw `<Summary>` tag only. A UID-0 project-summary row whose XML omits `<Summary>` but
carries a baseline finish leaks a project-spanning rollup baseline into the CPLI denominator. Real exports
usually set `Summary=1` on row 0 (hence Medium), but the two code paths disagree about what "summary" means.

### M5 — [MEDIUM] days↔% client/server rounding mismatch — VERIFIED
`web/app.py:5753` emits `SF_REMAIN_DAYS[uid] = round(rem/mpd, 3)`; `sra_risk.js` averages those rounded
per-task values, but the authoritative `_affected_avg_remaining_days` (`:5853`) averages the **unrounded**
`rem/mpd`. Negligible for multi-day tasks (lossless), but for sub-day remaining durations the derived
magnitude visibly diverges, and the JS-on path persists the client's value verbatim. Fix: round both sides
at the same point (emit higher precision, or round each per-task value before the server averages).

### M6 — [MEDIUM] `preserves_figures` is sign-blind — VERIFIED
`ai/citations.py:25` (`_FIGURE_RE = \d+(?:\.\d+)?`, no leading `-`). A model can turn `Variance -5 days`
(ahead) into `Variance 5 days behind` and the gate keeps it (both contain the token `5`). Sign is
load-bearing in schedule forensics (variance, float, slip direction). Fix: include an optional sign in the
regex so a flip changes the multiset and forces the verbatim fallback.

### M7 — [MEDIUM] `freezeColumns` per-cell style writes + no filter debounce — VERIFIED (perf)
`web/static/gantt.js:132-158`, called from `path.js` on every `paintRows()`; the `pathFilter` `input`
handler (`path.js:404`) calls `paintRows()` with no debounce. On the ~1700-row grid with ~6 data columns
that is ~10k inline-style writes + a full `tbody` rebuild per keystroke — typing in the path filter janks.
Fix: debounce the filter (120–150 ms) and/or cache last offsets and bail when unchanged.

### M8 — [MEDIUM] MEI/BEI can exceed 100% — VERIFIED (intentional for BEI, undocumented for MEI)
`engine/metrics/completion_performance.py:185-212` (`_mei`); same shape `dcma14.py:264-303` (`compute_bei`).
`_mei` counts **all** milestones finished by status in the numerator but only baselined-due in the
denominator, so a milestone finished-by-status but baselined-after-status produces num>den (verified
`value=2.0`). For BEI this is the documented Acumen formula; for MEI the >100% reading is defensible but not
called out. Fix: add a one-line help note that the index can exceed 1.0 — no math change.

### Low / Nit (one line each)
- **L1** `float_analysis.critical_incomplete_count` (pure CPM `is_critical`) vs `schedule_quality` (`is_effective_critical`): two "Critical" counts can silently disagree on a progressed file. Route both through `is_effective_critical` or label the two surfaces distinctly.
- **L2** `inf`/`nan` magnitudes accepted at entry (`_to_float` accepts them) then 422 every later SRA run until removed. Reject with `math.isfinite` at entry. (No 500 — the sim routes return a clean 422.)
- **L3** `shutdown_offload()` is wired only to `POST /api/shutdown`; the browser-gone watchdog exit skips it. Mitigated by `ProcessPoolExecutor`'s own atexit join; add a FastAPI shutdown/atexit hook for completeness.
- **L4** `derive()` returns early on `avg<=0`, leaving a stale derived value that then posts as "locked." Clear the unlocked field instead.
- **L5** DCMA-08 thresholds on the *current* `duration_is_elapsed`, not the baseline duration being tested (no `baseline_duration_is_elapsed` exists). Tiny impact; document the approximation.
- **L6** EVM BCWS is finish-gated (whole budget credited iff `baseline_finish ≤ status`), not a time-phased PV curve. Not wrong under scope (goldens aren't cost-loaded); flag for a future cost-loaded parity target.
- **L7** A non-integer cosmetic field (e.g. `OutlineLevel=n/a`) refuses the whole file via `_int`. Use a tolerant default-0 parse for cosmetic-only fields; keep the loud `_int` for identity/structural fields.
- **L8** `freezeColumns` left-offsets come from header `offsetWidth`; latent coupling to `table-layout:fixed` (currently satisfied because `SFColResize.attach` pins widths first). Document the precondition.
- **L9** `sra_risk.js` derive math is only substring-asserted, never executed (only `node --check` runs). Add a tiny node-driven unit harness against the same cases as the server test, or accept "server is authoritative" explicitly.
- **L10** `test_offload.py:97-101` asserts exact call spelling + threshold text; a harmless refactor breaks it with no behavior change. Prefer a behavioral assertion (monkeypatch `run_maybe_offloaded`, drive a >threshold schedule, assert `offload=True`).
- **L11** MSPDI duplicate links differing only by lag are collapsed order-dependently (dedup by `(pred,succ,type)`). Matches the XER importer; negligible real-world impact.
- **L12** `pyproject.toml:116` `fail_under=99.9` is **dead/misleading** config: `addopts` (`:99`) has no `--cov`, so local `pytest -q` enforces **no** coverage gate at all, and CI's command-line `--cov-fail-under=70` (`ci.yml:60`) overrides the pyproject value. The real gates are CI-only — overall ≥70%, engine ≥85% (`ci.yml:63`). *Note:* CLAUDE.md's "engine ≥85%, overall ≥70%" is **accurate**; the misleading number is the pyproject `99.9`. Set it to the real value (70.0) or remove it, and optionally add `--cov` to `addopts` so a coverage regression fails locally, not only in CI.
