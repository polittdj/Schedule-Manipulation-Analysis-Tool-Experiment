## HIGH (3)

### H1 — Translation path emits AI figures with NO figure gate (contradicts a documented invariant)
- **Where:** `web/app.py:875-899` (`_ai_translate`; `generate` at :891, stored verbatim :898); route
  :5161; client `web/static/translate.js`.
- **What:** CLAUDE.md states "narrative / briefing / **translation** … re-verify every AI-emitted
  figure against engine citations." `_ai_translate` does not — it sends prose to the local model with
  only a *soft* "keep numbers unchanged" instruction and stores the reply verbatim. It is the **only
  ungated** `.generate()` of five (qa/narrative/briefing/`_polished_narrative` all gate).
- **Impact (Law 2):** with a live local model + a non-English UI language, a rendered figure ("12 of
  126") can come back corrupted ("12 de 120") and replace the DOM text with no gate and no
  "AI-translated — verify" marker; it also re-translates an already-gated strict Q&A answer ungated at
  the display layer. Default `NullBackend` returns `{}`, so this needs a live model to trigger.
- **Validated:** (1) `sed` of `_ai_translate` shows no `reattach`/`preserves_figures`; (2) it is the
  one ungated emission of the five `.generate(` sites; (3) contradicts CLAUDE.md's explicit claim;
  (4) no test asserts figure preservation through translation.
- **Fix direction:** gate each translated line with `citations.preserves_figures(src, translated)`;
  fall back to the source line on mismatch (or mask→restore figures verbatim). Add a regression test.

### H2 — USER-GUIDE tells users the default Q&A mode is the *ungated* "interpretive" mode
- **Where:** `docs/USER-GUIDE.md:115,183` vs `web/app.py:5011` (`qa_mode: str = Form("annotate")`).

### H3 — 24-hour (continuous-ops) calendars silently parsed as 8h/day × 7 days
- **Where:** `importers/_common.py:199` (`working_time_span`) + `importers/mspdi.py:389,396,410`.
- **What:** An MS-Project "24 Hours" calendar encodes each day as a single `WorkingTime`
  `00:00:00 → 00:00:00`. `working_time_span("00:00:00","00:00:00")` returns `None` → 0 minutes for the
  day → `minutes_per_day` falls back to **480**; but `work_weekdays.add()` runs *before* the
  `minutes>0` gate, so all 7 weekdays still count. Result: a 24h calendar reports **7 days × 8 h**
  instead of 1440 min/day. Every CPM date, float, day-conversion, and DCMA "> 44 working days"
  threshold is then computed on the wrong axis. The tool explicitly targets outage/turnaround work
  (`TP3_Outage` fixture) — exactly the 24h-calendar domain.
- **Validated:** (1) `working_time_span("00:00:00","00:00:00")==None` vs `"…","24:00:00"==(0,1440)`
  (executed); (2) end-to-end a 7-day `00:00→00:00` calendar parses to `working_minutes_per_day==480`,
  all-7 weekdays; (3) a `PT48H` task = 2 days on a real 24h calendar but 6 "days" here; (4) untested
  (tests cover 10h, holidays, `DayWorking=0` dead calendars — never `DayWorking=1` + `00:00→00:00`).
- **Fix direction:** in `working_time_span`, treat `from==0 and to==0` as `(0,1440)` (a genuine
  non-working day is `DayWorking=0` with no `WorkingTime`, so `00:00→00:00` is unambiguously 24h);
  optionally move `work_weekdays.add` behind the `minutes>0` gate. **Verify the `.mpp`/MPXJ path**
  first — MPXJ may emit `24:00` (which already parses correctly), in which case only the direct-XML
  path is affected.

### M4 — Number-words bypass every figure gate (tokenizer is digits-only)
- **Where:** `ai/citations.py:31,38,41-49` (`figure_tokens`).
- **What:** The tokenizer matches only `-?\d+(\.\d+)?` + ISO dates. A spelled-out number ("twelve",
  "twenty") is invisible, so a rephrase that *introduces* a number-word passes `preserves_figures`, and
  a strict-mode Q&A answer whose only quantitative claim is a number-word passes `_classify_figures`
  clean → an unsourced count reaches the analyst in the mode that promises none does.
- **Validated:** (1) `figure_tokens("Twelve activities")==[]` (executed); (2)
  `preserves_figures(no-digit src, "+twelve")==True` (executed); (3) strict path finds zero tokens →
  not discarded; (4) no test uses a number-word. (Substitution digit→word is still safe.)
- **Fix direction:** extend `figure_tokens` (the single shared tokenizer) with a bounded number-word
  lexicon; treat an introduced number-word like an introduced digit.
